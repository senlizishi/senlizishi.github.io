Calcite之VolcanoPlanner优化过程解析

2021-01-29 大数据技术 Calcite 评论

原文地址：

https://zhuanlan.zhihu.com/p/283362100

导读：VolcanoPlanner是calcite自带的基于代价的优化器。在优化过程中，会计算优化规则生成的等价关系表达式的计算代价，通过动态规划算法找出整个最优的执行计划。

使用VolcanoPlanner

这里以一个单元测试为例子解释如何使用calcite的VolcanoPlanner。

@Test
  void myTest() {
    ConnectionSpec SCOTT = CalciteAssert.DatabaseInstance.HSQLDB.scott;
    SchemaPlus rootSchema = Frameworks.createRootSchema(true);
    Map<String, Object> map = new HashMap<>();
    map.put("jdbcDriver", SCOTT.driver);
    map.put("jdbcUser", SCOTT.username);
    map.put("jdbcPassword", SCOTT.password);
    map.put("jdbcUrl", SCOTT.url);
    map.put("jdbcCatalog", SCOTT.catalog);
    map.put("jdbcSchema", SCOTT.schema);
    JdbcSchema schema = JdbcSchema.create(rootSchema, "JDBC_SCOTT", map);
    rootSchema = rootSchema.add("  JDBC_SCOTT", schema);
    FrameworkConfig config = Frameworks.newConfigBuilder()
        .defaultSchema(rootSchema)
        .parserConfig(SqlParser.Config.DEFAULT)
        .programs(Programs.standard()).build();


    RelBuilder relBuilder = RelBuilder.create(config);
    RelNode relNode = relBuilder.scan("EMP")
        .project(relBuilder.field("DEPTNO"), relBuilder.field("ENAME"))
        .sort(relBuilder.field("DEPTNO")).build();

    VolcanoPlanner planner = (VolcanoPlanner) relNode.getCluster().getPlanner();
    //获取期望的RelTraiset，这里是将Convention.None替换成EnumerableConvention
    RelTraitSet desired = relNode.getTraitSet().replace(EnumerableConvention.INSTANCE).simplify();
    //设置根节点，会从根节点开始迭代将所有子节点也注册到planner中
    planner.setRoot(planner.changeTraits(relNode, desired));
    RelNode result = planner.chooseDelegate().findBestExp();
  }

calcite提供了测试的框架，可以让我们直接构造关系表达式。VolcanoPlanner优化的是关系表达式。但是calcite也提供SqlParser和SqlToRelConverter，可以进行sql解析并且将sqlNode转换成关系表达式RelNode,在这个测试中不使用SqlParser和SqlToRelConverter来解析sql字符串生成关系表达式,而是直接手动创建逻辑关系表达式,所以使用calcite框架时也可以跳过sql解析与关系表达式转换而用其他方法生成逻辑表达式而直接使用calcite最核心的关系表达式优化功能。 在这个单元测试例子中生成的逻辑关系表达式为：

LogicalSort(DEPTNO)
    LogicalProject(DEPTNO,ENAME)
        JdbcTableScan(EMP)

以上使用的表EMP是calcite源码单元测试中已经存在的.

该关系表达式的等价sql为：

select DEPTNO,ENAME from EMP sort by DEPTNO

逻辑关系表达式不是物理算子不可以执行，所以在volcanoPlanner在执行优化之前要设置期望的RelTraitSet，比如将逻辑表达式的Convention转成EnumerableConvention（Enumerable算子是calcite自带算子实现了相应的算子算法），这样优化过后的算子才是可执行的。

在上面例子中逻辑表达式原来的RelTraitSet的Convention是Convention.NONE,期望转换的CONVENTION设置为EnumerableConvention就可以转换成Enumerable算子为根节点的可执行关系表达式

还需要通过setRoot设置优化器的根节点，在这时候会对算子进行注册，从根节点开始一直到所有子节点都会注册到planner。 后面会详细解析注册过程，整个注册过程就是将关系表达式转换成RelSubset和RelSet结构。

  • RelSet: 关系表达式的等价集合，在它的字段rels集合中的关系表达式都有相同的语义。它的subsets集合中的subset会记录有相同物理属性(traitSets)的关系表达式的最小代价RelNode。
  • RelSubset：记录有相同物理属性（记录在traitSet中）的关系表达式的最优RelNode。它也是一种RelNode，set字段记录它所属的RelSet

关系表达式的注册过程

在上面的单元测试例子中,在planner.setRoot之前要调用planner.changeTrait.这个方法是将关系表达式注册到优化器,经过这个方法后关系表达式RelNode的结构中会出现RelSet和RelSubset,代表关系表达式结构中各个层次之间等价关系表达式.

public RelNode changeTraits(final RelNode rel, RelTraitSet toTraits) {
    assert !rel.getTraitSet().equals(toTraits);
    assert toTraits.allSimple();
    //对传入的关系表达式进行注册，经过注册后关系表达式结构都变成RelSubset注册的结构
    RelSubset rel2 = ensureRegistered(rel, null);
    //如果期望的TraitSet与原来关系表达式的TraitSet一致
    //那么可以直接返回，但是比如要从逻辑表达式的TraitSet转成
    //Enumerable的TraitSet这里就会是false
    if (rel2.getTraitSet().equals(toTraits)) {
      return rel2;
    }
    //生成新的Subset这个Subset的TraitSet就是期望的TraitSet
    //后面也会作为planner的根节点
    return rel2.set.getOrCreateSubset(
        rel.getCluster(), toTraits, true);
  }

在上面changeTraits方法中核心的部分就是调用ensureRegistered方法进行注册.在ensureRegistered方法的开头会调用getSubset方法检查要注册的RelNode是否已经存在映射的RelSubset,如果存在说明已经注册并且传入的等价表达式不为空就会进行merge合并等价表达式加入同一个等价表达式集合.在最开始逻辑表达式注册时可定是没有注册的那么就会返回null，继续调用register方法进行注册，而在register方法中实际调用registerImpl方法进行注册。

registerImpl方法

在registerImpl方法中会经历一下几个步骤：

(1) 开头进行一些必要的检查，比如要注册的RelNode的traitSet的size是否与planner traitDefs的大小相同，否则抛出异常。
(2) 调用要注册的RelNode的onRegister方法，对于逻辑关系表达式这个方法会以深度优先的方式重复上面的注册过程，让子节点向完成注册，当所有子节点注册完后返回当前节点才继续下面的步骤。比如例子中的逻辑表达式，在这个方法中会递归直到JdbcTableScan算子没有其他子节点了继续registerImpl后面步骤，结束后然后再返回LogicalProject,最后返回LogicalSort。

LogicalSort(DEPTNO)
        LogicalProject(DEPTNO,ENAME)
            JdbcTableScan(EMP)

(3) 判断当前的RelNode是否Converter.如果rel是converter.那么converter会与它的子表达属于同一个RelSet。那么converter的set就从它的input通过getSet获取，不用创建新的。
(4) 如果当前RelNode对应的RelSet为空，创建一个新的RelSet对象，并且加入到优化器的allSets中。
(5) 调用registerClass方法。在这个方法中会获取RelNode的class对象，并且添加到planner的classes集合中，如果集合中不存在这个class会调用onNewClass方法。在VolcanoPlanner的onNewClass方法实现中首先会调用父类的onNewClass方法，在这时会回调relNode的register方法（这时如果我们自定义了一些关系表达式可以利用这一点给优化器添加自定义的规则等）。这个方法核心代码片段如下：

final Class<? extends RelNode> clazz = node.getClass();
        for (RelOptRule rule : mapDescToRule.values()) {
          if (isPhysical && rule instanceof TransformationRule) {
            continue;
          }
          for (RelOptRuleOperand operand : rule.getOperands()) {
            if (operand.getMatchedClass().isAssignableFrom(clazz)) {
              classOperands.put(clazz, operand);
            }
          }
        }

这部分代码是根据优化规则要匹配的关系表达式类与当前关系表达式的类是否匹配，如果匹配放入calssOperands集合（MultiMap结构）将关系表达式类与优化规则进行映射，这样只有类匹配的优化规则才会应用于对应的关系表达式。 在registerClass方法最后会将RelNode的convention放入conventions集合，如果是新的convention那么就会回调node.getConvention().register(this);方法。比如JdbcTableScan算子的Convention是JdbcConvention是一个新的convention,那么这个时候就会回调JdbcConvention的register方法,实现如下就是将jdbcRule添加到优化器中。

@Override public void register(RelOptPlanner planner) {
        for (RelOptRule rule : JdbcRules.rules(this)) {
          planner.addRule(rule);
        }
        planner.addRule(CoreRules.FILTER_SET_OP_TRANSPOSE);
        planner.addRule(CoreRules.PROJECT_REMOVE);
      }

(6) 调用addRelToSet将当前注册的RelNode加入对应的RelSet中，该方法返回RelSubset。在该方法中会通过RelNode的RelTraitSet判断RelSet是否存在对应的RelSubset，不存在创建一个新的并加入到RelSet的subsets集合中。将RelNode与对应的RelSubset放入优化器的mapRel2Subset集合中。最后会调用RelSubset的propagateCostImprovements方法根据注册的RelNode计算它的最优代价，与最优的关系表达式算子。比如JdbcTableScan注册时对应的RelSubset的计算出来的结果如下：

JdbcTableScan会去数据库计算它的真实代价，得到真实代价会比inf小，那么这里JdbcTableScan就会称为RelSubset的best计划。 对于Logicalxxx算子这里计算的代价都是infinity，因为convention.none的都是。在后面的优化中如果有等价的新的关系表达式节点新注册进来时，有更小的代价时这个RelSubset就会更新。这也是VolcanoPlanner基于成本优化的关键点。
(7) 调用fireRules方法。

void fireRules(RelNode rel) {
        for (RelOptRuleOperand operand : classOperands.get(rel.getClass())) {
          if (operand.matches(rel)) {
            final VolcanoRuleCall ruleCall;
            ruleCall = new DeferringRuleCall(this, operand);
            ruleCall.match(rel);
          }
        }
      }

将前面classOperands中候选的operand拿出来，根据operand的结构与relNode的结构比较，如果匹配，那么Operand的rule就作为候选的rule放入ruleQueue。因为注册的时候是深度遍历优先，所以在ruleQueue中前面的rule都是RelNode表达式中最底层表达式匹配的rule开始的，实现了从下到上的规则匹配。关于具体的规则匹配解释可以参考Apache Calcite的优化器规则解析。

小结

逻辑关系表达式在通过VolcanoPlanner优化之前需要先将关系表达式各个节点注册到优化器中。注册方法采用深度优先遍历，首先注册子节点生成Relset关系表达式等价集合与注册RelNode对应的RelSubset计算最小代价信息。在注册关系表达式节点RelNode时会根据已有的优化器规则对关系表达式结构进行匹配，符合匹配规则的优化规则放入优化器的规则队列中。

在注册后，单元测试例子中的逻辑关系表达式结构变换如上图。结构上加入了RelSubset与Relset构成了各个层级的等价表达式集合与每个层次最小代价信息。后面在执行VolcanoPlanner优化器算法时就是利用各种优化规则对关系表达式进行变换，生成新的关系表达式节点加入对应的等价集合层次，利用动态规划的算法不断更新最小代价信息。最后优化过程结束后就可以根据每层的最小代价信息获得代价最小的可执行关系表达式路径。

优化

Calcite优化器执行优化算法的方法为findBestExp。VolcanoPlanner的该方法实现代码如下：

public RelNode findBestExp() {
    ensureRootConverters();
    registerMaterializations();

    //核心部分，从规则队列取出优化规则对表达式进行变化
    ruleDriver.drive();

    if (LOGGER.isTraceEnabled()) {
      StringWriter sw = new StringWriter();
      final PrintWriter pw = new PrintWriter(sw);
      dump(pw);
      pw.flush();
      LOGGER.info(sw.toString());
    }
    dumpRuleAttemptsInfo();
    // 通过所有规则变化后，现在所有可行的关系表达式路径都生成了
    // 在这里根据每一层等价表达式的RelSubset找出最小代价的关系表达式节点
    // 每一层最小代价表达式结合起来就构成了最优的可执行的关系表达式
    RelNode cheapest = root.buildCheapestPlan(this);
    if (LOGGER.isDebugEnabled()) {
      LOGGER.debug(
          "Cheapest plan:\n{}", RelOptUtil.toString(cheapest, SqlExplainLevel.ALL_ATTRIBUTES));

      if (!provenanceMap.isEmpty()) {
        LOGGER.debug("Provenance:\n{}", Dumpers.provenance(provenanceMap, cheapest));
      }
    }
    return cheapest;
  }

优化中的规则执行

一般VolcanoPlanner的RuleDriver实现类是IterativeRuleDriver，它的drive方法源码如下：

@Override public void drive() {
    while (true) {
      LOGGER.debug("PLANNER = {}; COST = {}", this, planner.root.bestCost);

      VolcanoRuleMatch match = ruleQueue.popMatch();
      if (match == null) {
        break;
      }

      assert match.getRule().matches(match);
      try {
        match.onMatch();
      } catch (VolcanoTimeoutException e) {
        LOGGER.warn("Volcano planning times out, cancels the subsequent optimization.");
        planner.canonize();
        break;
      }

      // The root may have been merged with another
      // subset. Find the new root subset.
      planner.canonize();
    }
  }

上面的drive方法实现也很简单易懂，就是不断的从规则队列中取出优化规则并执行。这个方法退出条件有两种：1.规则队列没有规则需要执行了。2.在规则执行时抛出VolcanoTimeoutException异常，当VolcanoPlannerd的cancleFlag设置为true时会抛出这个异常，所以在使用如果觉得优化时间太长，超过一定时间可设置cancleFlag为true强制优化结束。

下面解释一下在这里规则是如何执行的并且规则生成新的关系表达式后做了些什么？ 在我们上面测试例子中，VolcanoPlanner的规则队列中会有一个VolcanoRuleMatch，这个VolcanoRuleMatch的rule是
JdbcToEnumerableConverterRule,作用的关系表达式列表中只有JdbcTableScan。以这个为例进行解释。 在上面的代码match.onMatch调用中会调用getRule().onMatch(this)执行规则的onMatch方法。
JdbcToEnumerableConverterRule的onMatch方法实现复用ConverterRule实现：

@Override public void onMatch(RelOptRuleCall call) {
    RelNode rel = call.rel(0);
    if (rel.getTraitSet().contains(inTrait)) {
      final RelNode converted = convert(rel);
      if (converted != null) {
        call.transformTo(converted);
      }
    }
  }

JdbcToEnumerableConverterRule`的onMatch方法就是生成新的RelNode converted。该RelNode结构如下：

JdbcToEnumerableConverter
    JdbcTableScan

通过transformTo方法将新的RelNode结构注册到优化器中。在transformTo方法中最终还是会调用RelSubset subset =
volcanoPlanner.ensureRegistered(rel, rels[0]);通过ensureRegistered来注册。注册过程就是上一节描述的注册过程，只是这里equivRel等价关系表达式不在是null。在这个例子中要注册的新的RelNode是上面生成的JdbcToEnumerableConverter,等价关系表达式equivRel为原来的JdbcTableScan，因此这个注册过程JdbcToEnumerableConverter所在的等价集合就是JdbcTableScan所在的RelSet，在这个RelSet中会加入JdbcToEnumerableConverter对应的RelSubset记录着相应的最小代价信息。

transformTo方法后JdbcTableScan的等价集合RelSet的变化如上图所示。经过
JdbcToEnumerableConverterRule变换后虽然生成了新的关系表达式结构JdbcToEnumerableConverter，但是其最小代价大于原来的JdbcTableScan，所以这时这一层最优的关系表达式算子仍然是JdbcTableScan。 其他的规则与上面的例子大致一样，通过匹配已有的关系表达式结构生成新的等价关系表达式结构，放入等价关系表达式集合，根据新的最小代价信息看是否发现了更小的代价，上一层（parent）计算最小代价时会基于这一层的最小代价（动态规划）。

查找最优执行计划路径

在优化完成所有可执行的关系表达式路径都生成后，会使用buildCheapestPlan找出最优（代价最小）的RelNode结构路径。

RelNode buildCheapestPlan(VolcanoPlanner planner) {
    CheapestPlanReplacer replacer = new CheapestPlanReplacer(planner);
    final RelNode cheapest = replacer.visit(this, -1, null);

    if (planner.getListener() != null) {
      RelOptListener.RelChosenEvent event =
          new RelOptListener.RelChosenEvent(
              planner,
              null);
      planner.getListener().relChosen(event);
    }

    return cheapest;
  }

查找方法就是使用CheapestPlanReplacer采用visitor模式从上到下找到每一层最优算子组合起来。

public RelNode visit(
        RelNode p,
        int ordinal,
        RelNode parent) {
      if (p instanceof RelSubset) {
        RelSubset subset = (RelSubset) p;
        // 当前层次最由关系表达式节点
        RelNode cheapest = subset.best;
        . . . . . .
        p = cheapest;
      }
      . . . . . .
      List<RelNode> oldInputs = p.getInputs();
      List<RelNode> inputs = new ArrayList<>();
      // 继续处理子节点
      for (int i = 0; i < oldInputs.size(); i++) {
        RelNode oldInput = oldInputs.get(i);
        RelNode input = visit(oldInput, i, p);
        inputs.add(input);
      }
      if (!inputs.equals(oldInputs)) {
        final RelNode pOld = p;
        p = p.copy(p.getTraitSet(), inputs);
        planner.provenanceMap.put(
            p, new VolcanoPlanner.DirectProvenance(pOld));
      }
      return p;

总结

本文梳理VolcanoPlanner优化器优化过程中的一些关键地方，还有很多细节比较难以详细描述，Calcite中有很多概念也比较难以理解，要更好的理解VolcanoPlanner优化器优化过程还是需要跟着代码进行调试学习。在实际使用Calcite的时候知道优化过程有助于理解如何对Calcite优化，如何加入自己的优化规则，生成自定义的关系表达式算子等。
